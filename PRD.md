# Product Requirements Document (PRD): Программный комплекс по декомпиляции и глубокому аудиту безопасности Android APK

## 1. Цель проекта
Создание инструмента «замкнутого цикла» для перевода APK в исходный код (Java/Kotlin) с последующей автоматизированной проверкой безопасности и верификацией в песочнице. Ключевая цель — гарантировать, что декомпилированное приложение не только функционально идентично оригиналу, но и полностью безопасно.

## 2. Архитектура и Верификация

### 2.1 DEX to Source
*   **Описание**: Точная декомпиляция байт-кода Dalvik (DEX) в читаемый исходный код на Java или Kotlin.
*   **Инструменты**: Jadx, Fernflower.
*   **Требования**: Максимальное сохранение структуры проекта, ресурсов, манифеста и логики приложения. Обработка многомодульных APK (Multi-DEX).

### 2.2 Build-Back Identity
*   **Описание**: Механизм повторной сборки декомпилированного кода и побайтового сравнения с оригиналом.
*   **Требования**: Автоматическая генерация валидных `build.gradle` файлов. Использование тех же версий SDK/инструментов сборки. Цель - добиться максимальной идентичности (в идеале - побайтового совпадения критичных секций или идентичного поведения).

### 2.3 Closed Sandbox Testing
*   **Описание**: Сравнение поведения оригинального и собранного APK в изолированной среде на предмет функциональной идентичности.
*   **Среда**: Zero-Trust Sandbox (Docker-Android или специализированные эмуляторы).
*   **Метрики**: Сравнение логов исполнения, UI-состояний, сетевых запросов и файловых операций.

## 3. Модуль автоматизированного аудита безопасности (Security Scanning)

### 3.1 Static & Dynamic Analysis
*   **Статический анализ (SAST)**: Поиск уязвимостей в декомпилированном исходном коде. Инструменты: Semgrep, CodeQL.
*   **Динамический анализ (DAST)**: Поиск уязвимостей во время исполнения в песочнице. Инструменты: Frida, Xposed.

### 3.2 Инъекции
*   **Классические**: SQL-инъекции (SQLite), Command Injection, Path Traversal.
*   **Современные**: Prompt-инъекции (если приложение использует AI-модели/LLM, проверка отправляемых промптов и обработки ответов).

### 3.3 Контроль данных (Data Exfiltration)
*   **Описание**: Выявление несанкционированной передачи данных на внешние серверы, носители или скрытые сетевые запросы.
*   **Методы**: Перехват и анализ SSL/TLS трафика (MITM в песочнице), анализ использования API отправки данных, отслеживание подозрительных IP/доменов.

### 3.4 Доступ к ресурсам
*   **Описание**: Проверка на неправомерный доступ к конфиденциальным данным.
*   **Цели**: Контакты, камера, микрофон, файловая система, геолокация, внутренние идентификаторы устройства (IMEI, MAC, Android ID).
*   **Методы**: Статический анализ манифеста и вызовов API разрешений, динамическое отслеживание системных вызовов.

### 3.5 Malware Detection
*   **Описание**: Поиск сигнатур вредоносного ПО и подозрительной активности.
*   **Цели**: Backdoors, keyloggers, скрытые майнеры, трояны.
*   **Методы**: Эвристический анализ, анализ графа вызовов, интеграция с базами данных угроз (например, VirusTotal API для хэшей).

### 3.6 AI-Усиление (Local LLM Integration)
*   **Описание**: Использование локальных больших языковых моделей (LLM) для интеллектуального анализа найденных уязвимостей.
*   **Движок**: Ollama (запущена на хосте).
*   **Модели по умолчанию**: `qwen3.6:latest` или `gemma3:12b` (конфигурируется).
*   **Сценарии использования**:
    *   **Автоматическая фильтрация**: LLM анализирует фрагмент кода, где SAST нашел уязвимость, для отсеивания ложных срабатываний (False Positives).
    *   **Интерактивный ассистент**: В UI аналитик может запросить у ИИ подробное объяснение уязвимости и генерацию патча (рекомендации по исправлению).
    *   Абсолютная приватность: код не покидает локальный контур (Zero-Trust).

## 4. Функциональные требования
1.  **Security Scorecard**: Генерация подробного отчета о безопасности с рейтингом уязвимостей (CVSS), описанием рисков и рекомендациями по устранению. Выгрузка в PDF/JSON/HTML.
2.  **AI-Объяснения**: Возможность запросить текстовое объяснение и пример безопасного кода для каждой найденной уязвимости через локальную LLM.
2.  **Автоматическое развертывание**: Интеграция механизмов автоматического развертывания сборочного окружения (Gradle, Android SDK) для декомпилированного проекта.
3.  **Трассировка**: Встроенные инструменты трассировки системных вызовов в песочнице для детального анализа поведения (strace, ltrace-like функционал для Android).

## 5. Нефункциональные требования
1.  **Zero-Trust Sandbox**: Полная сетевая, файловая и процессная изоляция процесса анализа от основной ОС хоста. Использование эфемерных контейнеров/ВМ, которые уничтожаются после анализа.
2.  **Точность (False Positives)**: Минимизация ложных срабатываний при поиске уязвимостей за счет корреляции результатов статического и динамического анализа.

## 6. Технологический стек
*   **Движки декомпиляции**: Jadx, Fernflower, Apktool.
*   **Безопасность (SAST/DAST)**: Semgrep, CodeQL, Frida, Xposed Framework.
*   **Песочница**: Docker-Android, Android Emulator (AVD), Waydroid.
*   **Оркестрация/Backend**: Python (FastAPI/Celery) или Go для управления конвейером задач.
*   **Отчетность**: Vue/React для UI, генерация PDF (Puppeteer/WeasyPrint).

## 7. Риски и стратегии смягчения
*   **Обход анализа через динамическую подгрузку кода**: Динамический анализ должен перехватывать вызовы `DexClassLoader` и дампить подгружаемый код в память для последующего сканирования.
*   **Упаковщики (Packers)**: Использование специализированных инструментов для распаковки (Frida-скрипты для дампа памяти после распаковки) перед статическим анализом.
*   **Обфускация логики**: Применение деобфускаторов и анализа графа потока управления (Control Flow Graph) для восстановления читаемости логики. Использование AI для переименования переменных и методов (опционально).
